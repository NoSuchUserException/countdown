    Database Schema Design

    This should describe the design of your database schema. If your design does not use a database, mention it here.



    The database will store the information for our cells, entities (in this diagram listed as players), equipables, useables, and inventory. It is an integral portion of the system and contains the information necessary to generate encounters, items, and quests.
    Some Key Components Include:

  • Encounter will be a table consisting of at max 5 NPCs. NPC1 must be populated by an entity, NPC2-5 can be null.
  • Each cell will have a description by its [x][y] coordinate. A unique description or message will appear via the /INSPECT command. This is where quest relevant information will be stored.
  • Inventory table will contain the user’s inventory. At this time the user can hold 15 items. This can and probably will change with ease.
  • Inventory table will also contain the inventory of NPCs. NPCs will store significantly less items than the user’s inventory, but follow same format. Any items still in their inventory are drop loot in encounters. We have opted to “let the goblin drink” (an NPC can use the items in its inventory during battle if it chooses.)
  • Addendum: This will likely play a more serious role in later development as we tune our AI for the battles more than what currently exists.
  • Inventory, Equipable, and Useable are structured similarly to how the Encounter & Entity tables are structured. Finite number with tables existing to store the objects.
  • We have added the capacity for status ailments by having useable contain a “duration” time.

    Save And Load Use Case Diagram.

    On the menu page of the program, the user will be presented with a choice to load their game or start a new game. Related to these functions is also saving the game. The Save and Load functions call to local storage. Local storage returns the files. Once a file is successfully loaded in it will contact the database and interact with the client to generate the world. After this, gameplay begins.


    Cell Option Use Case Diagram

    When a player enters a cell, they are presented with options including travel, encounters, inspecting the environment, and taking items. The database stores information about the cells, including if an encounter exists and if items are present within a specific spot.
